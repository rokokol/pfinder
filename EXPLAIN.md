# Как работает решение

Документ описывает устройство утилиты поиска персональных данных в этом репозитории: как запускается сканирование, как извлекается текст, как находятся ПДн, как формируются `result.csv`, `result.json` и `result.md`, и зачем нужны дополнительные режимы вроде OCR, `suspicious` и `names-only`.

## Главная идея

Решение построено как конвейер:

```text
путь из CLI
  -> рекурсивный обход файлов
  -> извлечение текста по формату
  -> поиск ПДн и подозрительных кандидатов
  -> классификация уровня защищенности
  -> генерация отчетов
```

Основная команда:

```bash
uv run pii-scan "ПДнDataset/share" \
  --output reports/result.json \
  --csv-output reports/result.csv \
  --markdown-output reports/result.md \
  --workers 8 \
  --only-findings \
  --confidence strong \
  --ocr \
  --verbose
```

Для лидерборда главный файл - `reports/result.csv`. Он содержит только:

```csv
size,time,name
3068287,sep 26 18:31,CA01_01.tif
```

## Структура кода

- `src/pii_scanner/cli.py` - основной CLI `pii-scan`.
- `src/pii_scanner/scanner.py` - обход файлов, параллелизм, сбор результата по каждому файлу.
- `src/pii_scanner/extractors.py` - извлечение текста из разных форматов.
- `src/pii_scanner/patterns.py` - регулярные выражения, валидаторы и поиск ПДн.
- `src/pii_scanner/reports.py` - генерация CSV, JSON, Markdown и примерных отчетов.
- `src/pii_scanner/example_cli.py` - CLI `pii-example-report` для генерации примера отчетов из готового списка файлов.
- `tests/test_patterns.py` - быстрые тесты валидаторов, форматов отчетов и вспомогательных функций.

## CLI и параметры

`pii-scan` принимает файл или директорию:

```bash
uv run pii-scan INPUT_PATH [options]
```

Важные параметры:

- `--output` - путь к JSON-отчету.
- `--csv-output` - путь к CSV-отчету.
- `--markdown-output` - путь к Markdown-сводке.
- `--workers` - число параллельных обработчиков файлов.
- `--only-findings` - писать в отчеты только файлы с подтвержденными ПДн, suspicious-кандидатами или ошибками.
- `--ocr [LANG ...]` - включить OCR для изображений, embedded images внутри PDF и fallback OCR для проблемных PDF. Без языков используется default Tesseract.
- `--serial-ocr [LANG ...]` - включить последовательный OCR: каждый язык отдельно, затем все языки вместе.
- `--confidence weak|medium|strong` - минимальная уверенность подтвержденных находок для отчетов, по умолчанию `medium`.
- `--names-only` - в поле `name` писать только имя файла, без относительного пути.
- `--include-suspicious` - добавить suspicious-кандидатов в CSV/JSON. Для лидерборда этот флаг лучше не использовать.
- `--limit` - обработать только первые N файлов, удобно для smoke-проверок.
- `--max-pdf-pages` - ограничить количество страниц PDF.
- `--high-volume-threshold` - порог большого объема для классификации УЗ, по умолчанию 100000.

## Обход файлов

Обход реализован в `scanner.py`, функция `iter_files()`.

Если вход - файл, анализируется только он. Если вход - директория, используется рекурсивный обход `Path.rglob("*")`.

Служебные директории пропускаются:

```text
.git
.venv
__pycache__
.ipynb_checkpoints
```

Для каждого файла вызывается `scan_file()`.

## Параллелизм

Параллелизм включается параметром `--workers`.

В `scan_path()`:

- при `workers <= 1` файлы обрабатываются последовательно;
- при `workers > 1` используется `ThreadPoolExecutor`;
- каждая задача обрабатывает один файл через `scan_file()`.

Для сервера с 32 ГБ RAM разумный старт:

```bash
--workers 8
```

При OCR не стоит сразу ставить слишком большое значение: Tesseract и рендер PDF сильно нагружают CPU и память.

## Извлечение текста

Извлечение текста находится в `extractors.py`, главная функция - `extract_text()`.

Первое решение принимается по расширению файла:

```python
suffix = path.suffix.lower()
```

Дальше включается отдельная ветка под формат. Если ветка падает на конкретном файле, исключение перехватывается, в `warnings` добавляется описание проблемы, а сканирование остальных файлов продолжается.

Поддерживаемые форматы:

- `CSV` - через стандартный модуль `csv`;
- `JSON`, `TXT`, `MD`, `XML`, `LOG` - как текст;
- `HTML` - через `BeautifulSoup`;
- `RTF` - через `striprtf`;
- `PDF` - через `pypdf`, OCR embedded images и OCR fallback при необходимости;
- `DOCX` - через `python-docx`, затем XML fallback;
- `XLS/XLSX` - через `xlrd` и `openpyxl`;
- `Parquet` - через `pyarrow`;
- изображения `TIF/TIFF/JPG/JPEG/PNG/GIF` - через Tesseract, если включен `--ocr`;
- старый `.doc` и неизвестные текстовые файлы - best-effort fallback.

Результат извлечения:

```python
ExtractionResult(
    text=...,
    file_format=...,
    warnings=[...],
)
```

Ошибки отдельного файла не останавливают весь скан. Они превращаются в warning или error в результате файла.

## Логика по типам файлов

### Простые текстовые файлы

Расширения:

```text
.txt .md .json .xml .log
```

Файл читается как байты, затем декодируется функцией `_decode_bytes()`. Она последовательно пробует:

```text
utf-8
utf-8-sig
cp1251
latin-1
```

Если файл больше лимита `--max-bytes`, читается только первая часть, а в warnings добавляется сообщение о truncation.

Важно: JSON сейчас не парсится как структура. Он читается как текст. Это сделано потому, что для поиска ПДн достаточно текстового представления, а JSON в датасете может быть произвольной формы.

### CSV

CSV обрабатывается отдельно, не как обычный текст.

Алгоритм:

1. Файл декодируется через `_decode_bytes()`.
2. По первым 4096 символам `csv.Sniffer()` пытается определить диалект: разделитель, кавычки и т.п.
3. Если диалект не определился, используется стандартный `excel`.
4. `csv.DictReader` читает строки как словари `column -> value`.
5. Каждая строка превращается в текст с именами колонок.

Пример:

```csv
customer_name,email
Иванов Иван Иванович,ivan@example.com
```

становится:

```text
customer_name: Иванов Иван Иванович email: ivan@example.com
```

Это важно для полей вроде `destination_address`, `customer_name`, `passport`, потому что имя колонки дает контекст регуляркам и снижает шум.

Если указан `--max-rows`, после этого числа строк CSV обрезается, а в warnings пишется `CSV truncated`.

### HTML

HTML читается как текст и передается в `BeautifulSoup`.

Алгоритм:

1. Удаляются теги `script` и `style`.
2. Из DOM берется видимый текст через `soup.get_text("\n")`.
3. Если `BeautifulSoup` падает, используется грубый fallback: удаление HTML-тегов регуляркой.

Цель - не сохранить верстку, а получить текст, пригодный для поиска email, телефонов, адресов и других ПДн.

### RTF

RTF сначала декодируется как текст, потом передается в `striprtf.rtf_to_text()`.

Если библиотека не смогла разобрать файл, возвращается исходный текст. Это лучше, чем полностью потерять файл: в RTF часто остаются читаемые куски.

### PDF

PDF проверяется в несколько этапов.

Базовый путь:

1. `pypdf.PdfReader` открывает файл.
2. Если PDF зашифрован, код пробует `reader.decrypt("")`.
3. Для каждой страницы вызывается `page.extract_text()`.
4. Ошибки отдельных страниц записываются в warnings, но не останавливают весь PDF.

Отдельно перехватываются warning-сообщения `pypdf`. Для этого используется временный logging handler `_LogCapture`. Это нужно, чтобы поймать случаи вроде:

```text
Advanced encoding /SymbolSetEncoding not implemented yet
```

Если включен `--ocr` или `--serial-ocr`, после текстового извлечения PDF дополнительно проверяется на embedded images. Страницы с такими картинками отправляются в OCR, даже если обычный текст PDF извлекся нормально.

Если есть warnings `pypdf` или извлеченный текст выглядит плохим, при включенном OCR запускается PDF OCR fallback по всем страницам.

### Как определяется плохой PDF-текст

Функция `_looks_like_poor_pdf_text()` убирает пробелы и считает долю букв.

PDF-текст считается плохим, если:

- после удаления пробелов текста нет;
- или текст длиннее 80 символов, но букв меньше 25%.

Такая эвристика ловит случаи, когда PDF вернул много мусорных символов, цифр или управляющих последовательностей вместо нормального текста.

### PDF OCR fallback

Если включен `--ocr` или `--serial-ocr`, PDF OCR работает в двух режимах:

- для обычного PDF с embedded images OCRятся только страницы, где найдены картинки;
- для проблемного PDF с warning от `pypdf` или плохим текстом OCRятся все страницы.

Процесс OCR:

1. `PyMuPDF` открывает PDF.
2. Каждая страница рендерится в изображение с масштабом `2x`.
3. Изображение передается в Tesseract.
4. В обычном режиме без языков используется default Tesseract.
5. Если переданы языки, например `--ocr ru en`, используется один запуск `rus+eng`.
6. В режиме `--serial-ocr ru en` отдельно запускаются `rus`, `eng`, затем `rus+eng`.
7. OCR-текст объединяется с текстом, который смог извлечь `pypdf`.

Если `PyMuPDF` недоступен, используется fallback-рендер через `pypdfium2`. На текущем сервере это важно, потому что `PyMuPDF` может требовать системный `libstdc++.so.6`.

Это медленнее, но помогает для сканов, кривых шрифтов и PDF с нестандартными кодировками.

### DOCX

DOCX сначала читается через `python-docx`.

Извлекается:

- текст параграфов;
- текст ячеек таблиц.

Если `python-docx` падает, включается XML fallback:

1. DOCX открывается как ZIP.
2. Читается `word/document.xml`.
3. XML-теги удаляются регуляркой.
4. HTML entities декодируются через `html.unescape`.

Такой fallback грубый, но позволяет достать текст из части поврежденных или нестандартных DOCX.

### XLS и XLSX

Для `.xlsx` используется `openpyxl` в режиме `read_only=True`.

Алгоритм:

1. Открывается workbook.
2. Для каждого листа первая строка считается заголовками.
3. Остальные строки превращаются в `column: value`, как CSV.
4. В текст добавляется имя листа.

Для `.xls` используется `xlrd`:

1. Открывается workbook.
2. Первая строка листа считается заголовками.
3. Остальные строки превращаются в текст с колонками.

Ограничение `--max-rows` применяется на каждый лист.

### Parquet

Parquet читается через `pyarrow.parquet.read_table()`.

Алгоритм:

1. Таблица читается в `pyarrow`.
2. Берутся имена колонок.
3. Строки превращаются в словари.
4. Каждая строка форматируется как `column: value`.

Если `pyarrow` не может загрузиться из-за системных библиотек, например `libstdc++.so.6`, файл не роняет скан. В warnings пишется `Parquet extraction failed`.

### Изображения

Расширения:

```text
.tif .tiff .jpg .jpeg .png .gif
```

Если `--ocr` не указан, изображение не читается, а в warnings пишется:

```text
image skipped; pass --ocr to enable OCR
```

Если `--ocr` указан:

1. `PIL.Image.open()` открывает изображение.
2. Если языки не переданы, Tesseract запускается без параметра `lang`, то есть со своим языком по умолчанию.
3. Если языки переданы, они склеиваются через `+`, например `--ocr ru en` превращается в `lang="rus+eng"`.

Если указан `--serial-ocr ru en`, изображение читается тремя отдельными проходами: `rus`, `eng`, `rus+eng`. Если языки не переданы, выполняется один serial-вариант с default Tesseract.

### Старый DOC

`.doc` - старый бинарный формат Microsoft Word. Полноценного парсера для него сейчас нет.

Поэтому используется fallback `_extract_binary_strings()`:

1. Читаются байты.
2. Файл проверяется эвристикой `_looks_like_text_bytes()`.
3. Если байты похожи на текст, из них извлекаются печатные строки.
4. Если файл выглядит бинарным мусором, он пропускается.

Это best-effort режим. Он может найти текст в некоторых старых DOC, но не гарантирует полноту.

### Неизвестные расширения и файлы без расширения

Для неизвестных файлов сканер не доверяет расширению и проверяет сами байты.

Если байты похожи на текст, файл декодируется и из него вытаскиваются печатные строки. Если байты похожи на бинарник, файл пропускается.

Такой подход нужен для корпоративных хранилищ, где файлы могут быть без расширений, но при этом содержать текстовые данные.

## CSV с контекстом колонок

CSV не просто склеивается в строку. Каждая запись превращается в вид:

```text
customer_name: Иванов Иван Иванович email: ivan@example.com phone: +79991234567
```

Так рядом со значением сохраняется имя колонки. Это помогает находить адреса, ФИО и другие сущности точнее.

## Защита от бинарного мусора

Для неизвестных файлов без поддерживаемого расширения используется проверка `_looks_like_text_bytes()`.

Файл считается бинарным и пропускается, если:

- начинается с сигнатуры ELF, MZ, PNG, JPEG, GIF, PDF или ZIP;
- в первых 8192 байтах есть NUL-байт;
- доля печатаемых байтов меньше 85%.

Это нужно, чтобы файлы вроде `lost+found/#958615`, которые оказались ELF-бинарниками, не попадали в отчет из-за случайных строк из `strings`.

## OCR

OCR включается флагом:

```bash
--ocr
```

Более тщательный последовательный режим включается флагом:

```bash
--serial-ocr
```

Без списка языков `pytesseract.image_to_string(image)` вызывается без `lang`, поэтому используется default Tesseract.

Языки можно передать прямо после флага:

```bash
--ocr ru en
--serial-ocr ru en
```

Алиасы нормализуются:

```text
ru -> rus
en -> eng
```

Для `--ocr ru en` вызывается:

```python
pytesseract.image_to_string(image, lang="rus+eng")
```

В режиме `--serial-ocr ru en` используются три отдельных запуска Tesseract:

```text
rus
eng
rus+eng
```

Каждый результат попадает в `ExtractionResult.scan_texts`. Сканер запускает поиск ПДн по каждому варианту отдельно, затем объединяет категории. Для одинаковой категории берется максимальный счетчик среди OCR-вариантов, а не сумма, чтобы один и тот же номер или email не учитывался три раза.

Tesseract должен быть установлен в системе:

```bash
tesseract --version
tesseract --list-langs
```

Для `--ocr ru en` и `--serial-ocr ru en` в списке должны быть `rus` и `eng`. Если языки не передаются, используется default Tesseract.

## PDF и OCR fallback

PDF сначала читается через `pypdf`.

Если PDF содержит embedded images, страницы с этими картинками OCRятся при `--ocr` или `--serial-ocr`.

Если `pypdf` выдает warning вроде:

```text
Advanced encoding /SymbolSetEncoding not implemented yet
```

или извлеченный текст выглядит плохим, то при включенном OCR включается fallback по всем страницам:

```text
страница PDF -> изображение через PyMuPDF -> Tesseract OCR -> текст
```

Текст из `pypdf` и OCR объединяется. Это увеличивает шанс найти ПДн в картинках внутри PDF и в проблемных PDF, но замедляет скан.

При `--serial-ocr ru en` PDF OCR выполняется в буквальном порядке языков: сначала все выбранные страницы читаются как `rus`, затем как `eng`, затем как `rus+eng`. Если языки не указаны, используется один проход default Tesseract. Результаты не просто склеиваются перед поиском: детектор ПДн применяет регулярки и валидаторы к каждому OCR-варианту отдельно, затем объединяет найденные категории.

## Поиск ПДн

Поиск находится в `patterns.py`.

Каждая сущность описана через `PatternSpec`:

```python
PatternSpec(
    key="bank_card",
    label="банковская карта",
    kind="payment",
    regex=...,
    validator=is_luhn_valid,
    suspicious_on_failed_validation=True,
    confidence="strong",
)
```

Основные категории:

- email;
- телефон;
- ФИО;
- дата рождения;
- место рождения;
- адрес;
- паспорт РФ;
- СНИЛС;
- ИНН;
- водительское удостоверение;
- MRZ;
- банковская карта;
- банковский счет;
- БИК;
- CVV/CVC;
- биометрические данные;
- состояние здоровья;
- религиозные убеждения;
- политические убеждения;
- расовая или национальная принадлежность.

## Валидаторы

Для части сущностей есть строгие проверки:

- банковские карты - алгоритм Луна;
- СНИЛС - контрольная сумма;
- ИНН - контрольные цифры.

Если значение не проходит валидатор, оно не считается подтвержденной находкой.
Если валидатор прошел, дополнительный текстовый контекст для этой числовой сущности не требуется. Для банковских карт дополнительно проверяется платежный префикс, чтобы случайные длинные числа с валидной суммой Луна не попадали в `strong`.

## Уверенность находок

После извлечения текста каждая подтвержденная находка получает уровень `confidence`:

- `weak` - слабые признаки, которые часто бывают описанием категории, а не данными субъекта: `religion`, `political`, `race`, упоминания здоровья, БИК, расчетный счет, 10-значный ИНН юрлица;
- `medium` - обычные ПДн: ФИО, телефон, email, дата рождения, место рождения, строгий адрес проживания/регистрации;
- `strong` - валидированные идентификаторы и сильные признаки: паспорт, СНИЛС, MRZ, водительское удостоверение, валидный ИНН, банковская карта с алгоритмом Луна и проверкой платежного префикса.

Флаг `--confidence` фильтрует подтвержденные находки до записи отчетов:

```bash
--confidence weak    # максимум полноты, больше FP
--confidence medium  # дефолтный баланс
--confidence strong  # минимум FP, но часть TP может потеряться
```

Перед фильтрацией сканер дополнительно повышает уверенность контекстных находок:

- ФИО рядом с телефоном, email, датой рождения или адресом становится `strong`;
- специальные категории рядом с ФИО становятся `medium`, рядом с сильным документным идентификатором - `strong`;
- банковские реквизиты рядом с ФИО или документным идентификатором становятся `medium`.

Blacklist по именам файлов не используется. Публичные документы отсекаются только за счет слабой уверенности абстрактных слов и более строгих регулярных выражений.

## Suspicious-кандидаты

OCR может ошибиться в одной цифре. Например, реальный номер карты может стать похожим на карту, но не пройти алгоритм Луна.

Для таких случаев есть отдельная группа `suspicious`.

В нее попадают кандидаты, которые:

- совпали с регуляркой;
- должны были пройти контрольный валидатор;
- не прошли этот валидатор.

Сейчас suspicious считается для:

- банковских карт;
- СНИЛС;
- ИНН.

По умолчанию suspicious не попадает в `result.csv` и `result.json`, чтобы не ухудшать лидерборд ложноположительными строками. Но suspicious всегда виден в `result.md`.

Чтобы добавить suspicious в CSV/JSON:

```bash
uv run pii-scan "ПДнDataset/share" \
  --output reports/result_with_suspicious.json \
  --csv-output reports/result_with_suspicious.csv \
  --markdown-output reports/result_with_suspicious.md \
  --workers 8 \
  --only-findings \
  --ocr \
  --include-suspicious \
  --verbose
```

Расширенный CSV получает поля:

```csv
size,time,name,suspicious,suspicious_count
```

Такой файл нужен для анализа, а не для лидерборда.

## Классификация УЗ

Классификация находится в `classify_protection_level()`.

Логика:

```text
нет находок -> нет ПДн
специальные категории или биометрия и объем > 100000 -> УЗ-1
специальные категории или биометрия и объем <= 100000 -> УЗ-2
общедоступные или иные данные и объем > 100000 -> УЗ-3
общедоступные или иные данные и объем <= 100000 -> УЗ-4
```

Порог большого объема задается:

```bash
--high-volume-threshold 100000
```

В таблице задания порог задан по количеству субъектов. Утилита не восстанавливает уникальных субъектов, поэтому использует общее количество находок в файле как приближение.

Для лидерборда УЗ не попадает в `result.csv`, но используется в Markdown-сводке.

## Формат результата по файлу

Внутри программы каждый файл превращается в `FileResult`:

```python
FileResult(
    size=...,
    time="sep 26 18:31",
    path="relative/or/name.ext",
    file_format=".pdf",
    categories=[...],
    counts={...},
    confidence_counts={...},
    suspicious_categories=[...],
    suspicious_counts={...},
    protection_level="УЗ-4",
    examples={...},
    confidence_by_category={...},
    suspicious_examples={...},
    warnings=[...],
    error=None,
)
```

`size` и `time` берутся из `stat()` файла. `time` форматируется как:

```text
mon dd HH:MM
```

Пример:

```text
sep 26 18:31
```

## Отчеты

Отчеты генерируются в `reports.py`.

### result.csv

Основной файл для сдачи:

```csv
size,time,name
3068287,sep 26 18:31,CA01_01.tif
```

В него попадают только файлы с подтвержденными ПДн, прошедшими порог `--confidence`. Если `--include-suspicious` не указан, suspicious-кандидаты не добавляют строки в CSV.

### result.json

По умолчанию JSON содержит тот же submission-список:

```json
[
  {
    "size": 3068287,
    "time": "sep 26 18:31",
    "name": "CA01_01.tif"
  }
]
```

С `--include-suspicious` добавляются:

```json
{
  "suspicious": {"банковская карта": 2},
  "suspicious_count": 2
}
```

### result.md

Markdown нужен для анализа, не для лидерборда. Он содержит:

- общую статистику;
- распределение по УЗ;
- распределение по категориям;
- распределение по уверенности;
- suspicious validation failures;
- top files;
- top suspicious files.

## Имена файлов и --names-only

По умолчанию `name` содержит относительный путь от корня сканирования:

```text
Документы партнеров/rules.pdf
Архив сканы/z/zzz97c00/527802957+-2958.tif
```

Если добавить:

```bash
--names-only
```

то в `name` будет только basename:

```text
rules.pdf
527802957+-2958.tif
```

Важно: если в разных папках есть файлы с одинаковым именем, `--names-only` может схлопнуть различия. Если проверяющая система ожидает относительные пути, этот флаг использовать не нужно.

## Примерные отчеты из списка файлов

Для генерации примера отчетов без детекции есть команда:

```bash
uv run pii-example-report \
  "ПДнDataset/share/Архив сканы/z/zzz97c00/527802957+-2958.tif" \
  "ПДнDataset/share/Документы партнеров/rules.pdf" \
  --root "ПДнDataset/share" \
  --output reports/example.json \
  --csv-output reports/example.csv \
  --markdown-output reports/example.md
```

Она не ищет ПДн. Все переданные файлы считаются найденными. Это нужно только для демонстрации синтаксиса отчетов.

Можно передать список из файла:

```bash
uv run pii-example-report \
  --file-list files.txt \
  --root "ПДнDataset/share" \
  --output reports/example.json \
  --csv-output reports/example.csv \
  --markdown-output reports/example.md
```

## Безопасность

В Markdown-сводке и внутренних структурах примеры ПДн маскируются:

```text
ivan@example.com -> iv***@e***
4111 1111 1111 1111 -> 41***11
Иванов Иван Иванович -> И*** И*** И***
```

В submission CSV и JSON вообще нет значений ПДн. Там есть только:

```text
size,time,name
```

## Ограничения

Решение эвристическое. Оно не является юридически строгой экспертизой ПДн.

Основные ограничения:

- регулярки могут давать ложноположительные срабатывания;
- OCR зависит от качества скана;
- старые `.doc` читаются best-effort;
- Parquet зависит от `pyarrow` и системных библиотек;
- адреса часто шумные, особенно в юридических и логистических документах;
- suspicious-кандидаты требуют ручной проверки.
